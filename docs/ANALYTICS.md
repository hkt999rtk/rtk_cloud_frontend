# Promotion Site Analytics Specification

## 1. Objective

Build a lightweight, first-party analytics system for the Realtek Connect+ promotion website to measure:

- Traffic source at a coarse domain level.
- Page engagement.
- CTA conversion.
- Content effectiveness.

This system is for operational product-marketing insight only. It must not become a cross-site tracking, advertising, profiling, or user-identification system.

## 2. Privacy Position

Default posture:

- No third-party analytics service.
- No advertising pixels.
- No fingerprinting.
- No collection of names, email addresses, company names, precise location, or account identifiers.
- No full referrer URL storage.
- No persistent analytics identifier unless the site later adds an explicit analytics opt-in.

Important constraint:

Even without `user_id`, analytics events can still describe user behavior. A `session_id` plus page events and referrer can be tracking data. The implementation must treat analytics as privacy-relevant telemetry and keep it minimal.

## 3. Consent And Storage Strategy

MVP strategy:

- Use a short-lived in-memory session id for the current page lifecycle.
- Do not store analytics ids in `localStorage` or cookies by default.
- Do not use `sessionStorage` unless the privacy notice is updated to say so.
- If persistent sessions are needed later, add an explicit opt-in analytics preference before writing to `localStorage`, cookies, or equivalent browser storage.

Allowed without persistent browser storage:

- One page-load `page_view`.
- CTA clicks.
- Coarse scroll milestones.
- One engagement event after a threshold.

Not allowed in MVP:

- Cross-visit identity.
- Cross-domain identity.
- Device fingerprinting.
- Raw IP address storage.
- User-agent based fingerprint fields.
- Full URL capture with query strings.

## 4. Architecture

```text
Frontend inline JS
    |
    | POST /api/event
    v
Go backend validation and sanitization
    |
    v
SQLite data/analytics.db
    |
    v
Admin-only aggregate queries
```

Configuration:

- `ANALYTICS_ENABLED`: default `true`.
- `ANALYTICS_DATABASE_PATH`: default `data/analytics.db`.
- `ANALYTICS_RETENTION_DAYS`: default `90`.

When `ANALYTICS_ENABLED=true`, the public pages may emit the first-party analytics JavaScript described in this specification. The default enabled mode must remain privacy-friendly: no third-party scripts, no persistent browser storage, no raw IP storage, no full referrer URL storage, and no form field capture.

When `ANALYTICS_ENABLED=false`, the public pages must not emit analytics JavaScript and `/api/event` should return `404` or `204` without storing data.

## 5. Event Model

Server-owned fields:

- `id`: INTEGER primary key.
- `ts`: INTEGER Unix timestamp generated by the server.
- `event`: TEXT allowlisted event type.
- `page`: TEXT allowlisted canonical page key.
- `cta`: TEXT nullable allowlisted CTA key.
- `percent`: INTEGER nullable scroll milestone.
- `duration`: INTEGER nullable seconds.
- `variant`: TEXT nullable allowlisted experiment key.
- `referrer_origin`: TEXT nullable sanitized origin only.
- `session_id`: TEXT short-lived opaque id for current page lifecycle.
- `created_at`: TEXT server timestamp.

Client-submitted fields:

- `event`
- `page`
- `cta`
- `percent`
- `duration`
- `variant`
- `session_id`

Server-derived fields:

- `ts`
- `referrer_origin`
- `created_at`

The server must ignore unknown fields.

## 6. Event Types

Allowed event types:

- `page_view`
- `click_cta`
- `scroll`
- `engaged`

Example `page_view`:

```json
{
  "event": "page_view",
  "page": "home"
}
```

Example `click_cta`:

```json
{
  "event": "click_cta",
  "page": "home",
  "cta": "contact_us"
}
```

Example `scroll`:

```json
{
  "event": "scroll",
  "page": "home",
  "percent": 50
}
```

Example `engaged`:

```json
{
  "event": "engaged",
  "page": "home",
  "duration": 10
}
```

## 7. Validation Rules

Request validation:

- Maximum body size: 4 KB.
- Method: `POST` only.
- Content-Type: `application/json`.
- Event must be in the allowlist.
- Page must be a canonical page key, not a raw URL.
- CTA must be an allowlisted key when present.
- Scroll percent must be one of `25`, `50`, `75`, `100`.
- Duration must be one of `10`, `30`, `60` for MVP.
- Variant must be an allowlisted key or empty.
- Session id must be generated by the site JS, opaque, and limited to 64 characters.
- Reject or ignore malformed values without storing raw payloads.

Referrer handling:

- Read from the HTTP `Referer` header.
- Store only scheme and host, for example `https://example.com`.
- Drop path, query string, and fragment.
- If parsing fails, store empty string.
- Do not store search keywords or campaign query parameters from referrer URLs.

IP and user agent:

- Do not store raw IP address.
- Do not store full user agent.
- If abuse protection needs IP, use it transiently in memory for rate limiting only.

## 8. Database Schema

```sql
CREATE TABLE IF NOT EXISTS analytics_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts INTEGER NOT NULL,
  event TEXT NOT NULL,
  page TEXT NOT NULL,
  cta TEXT,
  percent INTEGER,
  duration INTEGER,
  variant TEXT,
  referrer_origin TEXT,
  session_id TEXT,
  created_at TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_analytics_events_ts
  ON analytics_events(ts);

CREATE INDEX IF NOT EXISTS idx_analytics_events_event_page
  ON analytics_events(event, page);
```

Runtime database files must stay out of git.

## 9. API

Endpoint:

```text
POST /api/event
```

Request:

```json
{
  "event": "click_cta",
  "cta": "contact_us",
  "page": "home",
  "session_id": "2f3f4b7c-ephemeral"
}
```

Response:

```json
{
  "status": "ok"
}
```

Response behavior:

- Valid event: `202 Accepted` or `200 OK`.
- Invalid event: `400 Bad Request`.
- Disabled analytics: `404 Not Found` or `204 No Content`, with no storage.
- Rate-limited event: `429 Too Many Requests`.

## 10. Frontend Rules

Frontend implementation:

- Use small inline JavaScript, no npm dependency.
- Emit analytics only when `ANALYTICS_ENABLED=true`.
- Generate an ephemeral session id in memory.
- Do not store the id in `localStorage` or cookies in MVP.
- Use canonical page keys rendered by the server, not `location.href`.
- Send events with `navigator.sendBeacon` when available; otherwise use `fetch` with `keepalive`.

Tracked events:

- `page_view` once after load.
- `click_cta` for selected CTA links/buttons.
- `scroll` once per milestone at `25`, `50`, `75`, `100`.
- `engaged` once after 10 seconds of visible page time.

Do not track:

- Form field values.
- Contact form message content.
- Full clicked URLs.
- Mouse movement.
- Keystrokes.
- Precise viewport/device fingerprint fields.

## 11. Abuse Protection

The API must include:

- Body size limit.
- Event allowlist.
- Field length limits.
- In-memory per-IP rate limit.
- Same-origin or strict CORS policy.
- `Cache-Control: no-store`.
- No logging of raw request bodies.

## 12. Retention

Default retention:

- Raw events: 90 days.
- Aggregated monthly metrics: optional, up to 24 months.

The application should include a cleanup routine or admin command that deletes raw events older than `ANALYTICS_RETENTION_DAYS`.

## 13. Metrics Queries

Conversion rate:

```sql
SELECT
  COUNT(CASE WHEN event = 'click_cta' THEN 1 END) * 1.0 /
  NULLIF(COUNT(CASE WHEN event = 'page_view' THEN 1 END), 0)
FROM analytics_events;
```

Top referrer origins:

```sql
SELECT referrer_origin, COUNT(*)
FROM analytics_events
WHERE event = 'click_cta'
GROUP BY referrer_origin
ORDER BY COUNT(*) DESC;
```

Scroll distribution:

```sql
SELECT percent, COUNT(*)
FROM analytics_events
WHERE event = 'scroll'
GROUP BY percent
ORDER BY percent;
```

CTA clicks by page:

```sql
SELECT page, cta, COUNT(*)
FROM analytics_events
WHERE event = 'click_cta'
GROUP BY page, cta
ORDER BY COUNT(*) DESC;
```

## 14. Privacy Notice Updates

Before enabling analytics, update `/privacy` in all supported locales to explain:

- Analytics is first-party and stored in SQLite.
- Event types collected.
- Referrer origin handling.
- Session id behavior.
- Retention period.
- No third-party analytics, advertising pixels, or fingerprinting.
- How users can contact the privacy address for questions.
- Production enablement for issue #49 stays blocked until the privacy notice updates in issues #52 and #54 are both complete.

If persistent browser storage is added later, add an analytics preference UI before enabling it.

## 15. MVP Scope

MVP includes:

- `ANALYTICS_ENABLED` flag, defaulting to `true` for ephemeral first-party analytics.
- `POST /api/event`.
- SQLite storage in `data/analytics.db`.
- Validation and sanitization.
- Ephemeral frontend tracking.
- Retention cleanup.
- Basic aggregate query helpers or admin-only reporting endpoint.
- Tests for default enabled mode, disabled mode, valid events, invalid events, referrer sanitization, retention cleanup, and no raw form value capture.

MVP excludes:

- Google Analytics.
- Third-party scripts.
- Persistent cross-visit session ids.
- A/B test assignment storage.
- User-level dashboards.
- Advertising attribution.
- IP geolocation.
- Device fingerprinting.
